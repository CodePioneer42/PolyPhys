# Chromosome 3D Idea To Model

## Overview
Convert a free-form chromosome 3D idea into a physically explicit modeling draft without collapsing the full workflow into one response. Treat chromosome 3D structure and folding as the problem domain, separate candidate underlying mechanisms from macroscopic phenotypes, and prefer explicit controls plus validation guardrails over fast conclusions.

The hierarchy is fixed:
- Problem domain: chromosome 3D structure and folding
- Underlying mechanisms: loop extrusion, bridging, and other necessary drivers
- Macroscopic phenotypes: condensation, clustering, domain formation, compartment-like behavior

`phase separation` is not accepted as a final mechanism label by default. Reframe it as a descriptive phenotype unless the user provides explicit thermodynamic evidence.

###[WORKFLOW BREAKPOINTS & STATE LOCK] (CRITICAL)
1. **Sequential Execution:** The modeling process MUST be strictly executed in separate, sequential stages: Mechanism Decomposition -> Minimal Model Building -> Observable Translation -> Software Recommendation.
2. **NO INFO-DUMPING (ABSOLUTELY FORBIDDEN):** You MUST NOT output content for multiple stages in a single response. Never generate the "Minimal Physical Model" or "Execution Outline" during the mechanism brainstorming phase.
3. **Mandatory Stop Point:** After executing the mechanism decomposition (presenting candidate mechanisms and degeneracy warnings), you MUST **STOP GENERATING** immediately. You MUST end your response with a direct question to the user, such as: *"Which of these candidate mechanisms do you prefer to validate first?"*
4. **User Confirmation Gate:** You are FORBIDDEN to trigger the minimal model building phase or proceed to the next step until the user explicitly replies and confirms their chosen path/mechanism.

## When to Use
- A researcher has a chromosome 3D structure or folding idea and wants a model-ready draft.
- The prompt describes condensation, clustering, compaction, or domain formation but the mechanism is unresolved.
- The user labels behavior as phase separation without thermodynamic evidence.
- A project needs explicit controls, orthogonal validation, and mechanism competition before simulation implementation.

Do not use this skill for generic genomics Q&A, broad literature review only, or full simulation pipeline automation.

## Workflow
1. `Explore project context` first: check files, docs, and any chromosome-related data the user already provided. Do not default to git history; only inspect git history when the user asks for it or the current task depends on branch history.
2. Reframe the idea in physically neutral language. Lead with phenotype and scope, not a mechanism conclusion.
3. Decide whether the modeling path should stay on a `physical mechanism polymer model`, a `data-driven polymer model`, or an explicit comparison between the two.
4. Ask clarifying questions before locking the model whenever key biological or physical assumptions are missing.
5. Ask `one multiple-choice question per message` when clarification is still needed. Keep the prompt narrow so the model becomes clearer instead of noisier.
6. Identify at least two candidate underlying mechanisms and emit a `Degeneracy warning` when multiple explanations remain plausible.
7. Keep the first-stage output limited to `Problem Framing`, `Competing Mechanisms`, `Grounding Notes`, and the confirmation gate question.
8. Pause simulation recommendation if the mechanism space is still unresolved.
9. After the user confirms the chosen mechanism path, handoff to `minimal-model-builder` instead of drafting later stages in the same response.
10. After the user approves the saved design state, write to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` and record which model path and which skills generated by the approved draft.

## Checklist
- Explore the current files, docs, and chromosome-related data before proposing a model path.
- Classify the request as `physical mechanism polymer model`, `data-driven polymer model`, or an explicit comparison.
- Ask one question at a time until the missing assumptions are narrow enough to draft.
- Default to a multiple-choice format with `2-4` options plus `Other` when clarification is still active.
- Emit `Competing Mechanisms` and a visible `Degeneracy warning` before any later-stage handoff.
- Keep `The Minimal Physical Model`, `Baseline Control`, `Orthogonal Validation`, and software choice deferred until the user confirms the mechanism path.
- If the draft names software later, verify the capability claim against current official docs before presenting it as settled.
- Do not write the design doc until the user approves the modeling draft.

## Workflow State Machine
### State 1: Context And Path Selection
- Read the current files, docs, and user-provided data.
- Identify data sources before parameter files.
- Use a multiple-choice question when one remaining ambiguity blocks the path choice.
- Exit gate: the model path is chosen or the remaining ambiguity is specific enough for one clarifying question.

### State 2: Mechanism Decomposition
- Produce the draft with `Problem Framing`, `Competing Mechanisms`, `Grounding Notes`, and a direct confirmation question.
- Keep unsupported mechanism claims downgraded to phenotype wording.
- Do not generate `The Minimal Physical Model`, `Baseline Control`, `Orthogonal Validation`, or any later-stage planning in this state.
- Exit gate: candidate mechanisms are explicit, the `Degeneracy warning` is visible, and the response ends at the confirmation gate.

### State 3: Approval And Transition
- Pause simulation recommendation when degeneracy still dominates.
- After the user confirms one mechanism path, handoff to `minimal-model-builder`.
- After the user approves the documented draft, persist the design doc and record the selected path plus generating skills.
- Treat the saved design doc as an active end node.
- Exit gate: the mechanism path is confirmed for handoff, or the design doc is approved and closed at the active end node.

## After This Stage
- If the modeling draft is approved, write the design doc to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`.
- If the draft is unresolved, ask one question.
- If candidate mechanisms are explicit but unconfirmed, ask: *"Which of these candidate mechanisms do you prefer to validate first?"*
- If the user confirms a mechanism path, handoff to `minimal-model-builder`.
- If the draft is coherent but not yet approved, present the draft and wait instead of writing files.
- If the design doc has been written, end the conversation and wait for a new explicit request.

## Self-Review
- `stage completeness`: did the current stage include every required section before moving on?
- `workflow breakpoint`: did the response stop after mechanism decomposition instead of leaking later-stage content?
- `unsupported leaps`: did any phenotype get promoted to a mechanism without evidence?
- `premature handoff`: did the skill jump to later-stage modeling or documentation before the exit gate was met?
- `validation split`: are fitting and orthogonal validation still separated once later stages begin?
- `parameter restraint`: were parameter files deferred until the modeling path was approved?
- `question format`: did clarification stay in multiple-choice form unless forced otherwise?
- `end-node discipline`: did the skill end the conversation after the design doc was written?

## Context Exploration
Before proposing a final model, inspect the current project state:
- check files, docs, and any chromosome-related data
- inspect chromosome-related data files if the user provided them
- decide whether the request is better framed as a `physical mechanism polymer model` or a `data-driven polymer model`
- identify data sources before parameter files; do not read detailed parameter files during proposal stage by default
- only inspect parameter files after the modeling path is approved and parameterization is actually needed

Use the data-driven branch when experimental matrices, tracks, or structure-derived inputs already dominate the problem statement. Use the physical mechanism branch when the primary question is causal mechanism, control design, or physical interpretation.

## Proactive Questions
When the user under-specifies the system, ask targeted questions that expose hidden assumptions rather than generic “tell me more” follow-ups.

Examples:
- Is the proposed interaction `ATP-driven` and dissipative, or `passive` and compatible with thermal equilibrium?
- Does the polymer `Persistence length` have a biological reference value that should anchor stiffness?
- Is the interaction sequence-specific, state-specific, or effectively homogeneous at the current coarse-graining level?
- Is the observed phenotype stable, transient, cell-cycle dependent, or perturbation dependent?

Use proactive questions to reduce ambiguity before the workflow advances to model construction.

Rule: ask clarifying questions one at a time. Use `one multiple-choice question per message` until the decision boundary is clean enough to proceed.

Default format:
- Ask a multiple-choice question with `2-4` concrete options.
- Include `Other` as the final escape hatch.
- Only use an open-ended question when the state cannot be cleanly discretized without distorting the science.

## Output Protocol
Every first-stage response must use the following mechanism-first draft.

### 1. Problem Framing
- Phenomenon of Interest
- Scope and scale
- Known constraints
- Neutral restatement of the idea
- Missing assumptions to resolve through proactive questions
- Chosen path: `physical mechanism polymer model` or `data-driven polymer model`

### 2. Competing Mechanisms
- Candidate mechanism A
- Candidate mechanism B
- Degeneracy warning when more than one mechanism can explain the phenotype
- Explicit downgrade of unsupported `phase separation` language to phenotype-only wording

### 3. Grounding Notes
- Consensus
- Contested
- Heuristic
- Software Capability Check

### 4. Confirmation Gate
- Direct question to the user
- One multiple-choice question if the ambiguity can be discretized
- Stop condition for this conversation: wait for the user to choose the mechanism path

## Deferred Later Stages
Do not generate these sections until the user confirms the mechanism path and the next skill handoff begins:
- `The Minimal Physical Model`
- `Base Polymer Model`
- `Baseline Control`
- `Experimental Perturbation`
- `Active Components Required`
- `Active Module Checklist`
- `Fitting Observables`
- `Orthogonal Validation`
- `Degeneracy-Breakers`
- `In Silico Perturbations`

Later-stage software notes must stay tied to the existing model and use official docs when capability claims are version-sensitive. If software is named later, classify the route as `package-native` or `package + custom extension`; do not default to an abstract custom-code fallback.

Pause simulation recommendation when high degeneracy remains. Ask whether to add new experimental constraints or proceed with two competing control models.

Grounding Notes are short. They label epistemic status; they do not become a full literature review.

`Software Capability Check` is also short. Use it only when the draft names a simulation package, engine, or implementation route in a later confirmed stage. Verify capability claims against current official docs when the answer depends on package support or version-sensitive features. Confirm that the proposed route is executable for the existing model.

## Quick Reference
| Situation | Required response |
| --- | --- |
| User says `phase separation` without proof | Reframe as condensation, clustering, or domain formation and ask for thermodynamic evidence before accepting the mechanism label |
| User already supplied data files | Explore project context first and consider a `data-driven polymer model` branch |
| Proposal stage only needs source triage | Identify data sources and defer detailed parameter files until the modeling path is approved |
| Core physical assumption is missing | Ask clarifying questions using `Proactive Questions` before locking the model |
| Clarification is still needed | Ask one question per message instead of emitting a large provisional draft |
| One phenotype admits multiple mechanisms | Add `Degeneracy warning` and list `Competing Mechanisms` before any later-stage handoff |
| Model proposal lacks a control | Mark `Baseline Control` as a deferred later-stage requirement rather than fabricating it early |
| Same data type is used for fit and validation | Replace validation with `Orthogonal Validation` from a different observable class in the later confirmed stage |
| Mechanism is still unresolved late in the draft | Pause simulation recommendation and offer two competing control models or request extra constraints |
| Active processes may matter | Carry `Active Components Required` and `Active Module Checklist` into the later confirmed stage rather than hiding the assumption |
| A named simulation package appears in a later confirmed stage | Verify the capability in official docs and label the route as `package-native` or `package + custom extension` |
| A clarification step is still needed | Ask one multiple-choice question with `2-4` options and `Other` |
| The design doc has been written | Treat it as the active end node and end the conversation |

## Common Mistakes
- Treating `phase separation` as a final mechanism without explicit thermodynamic evidence.
- Giving one favored mechanism without a `Competing Mechanisms` section.
- Generating `The Minimal Physical Model` before the user has selected a mechanism path.
- Smuggling `Baseline Control` or `Orthogonal Validation` into the mechanism brainstorming response.
- Omitting the `Active Module Checklist` when nonequilibrium effects may be central once later stages begin.
- Using the same observable family for both `Fitting Observables` and `Orthogonal Validation`.
- Recommending software before resolving whether major mechanistic degeneracy remains.
- Naming software support without checking current official docs when the claim is version-sensitive.
- Presenting contested claims as settled instead of labeling them in `Grounding Notes`.
- Skipping `Explore project context` even when the user already supplied chromosome-related data files.
- Reading detailed parameter files before deciding whether the request is data-driven or mechanism-driven.
- Saving the design doc before the user has approved the modeling draft.
- Asking several refinement questions at once instead of following the `one question per message` rule.
- Using open-ended clarification when a multiple-choice question would have kept the state cleaner.
- Continuing the conversation after the design doc has been written instead of treating that point as the active end node.
- Failing to save the final design to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`.
